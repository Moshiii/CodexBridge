# CodexBridge CLI 视觉识别与启动动效设计

## 1. 目标

这份文档只定义 CodexBridge CLI 启动区的视觉方案，不涉及具体代码实现。

本次设计目标有两个：

1. 定义一个稳定、可长期使用的静态实心 ASCII logo
2. 定义一个轻量、克制的 glitch 扫描效果，只作用于备注/副标题行

约束也很明确：

- 不能做成全屏 TUI
- 不能依赖复杂重绘逻辑
- 不能让动画干扰 `readline` 输入
- 不能影响普通结果输出的可读性

结论先定下来：

- 启动区使用“静态 logo + 信息卡 + 一次性轻 glitch”组合
- logo 保持稳定，不参与抖动、错位、闪烁
- glitch 只扫过一行较小文本，作为启动阶段的气质点缀

## 2. 设计原则

### 2.1 视觉原则

- logo 要厚重、实心、稳，不做线框风格
- 启动区整体气质要偏工具感，而不是赛博炫技
- 信息密度要低，留白要足
- 颜色只做强调，不靠大面积彩色取胜

### 2.2 动效原则

- 动效必须短
- 动效必须局部
- 动效必须可关闭
- 动效结束后要停在稳定画面

### 2.3 工程原则

- 非 TTY 环境自动降级为纯静态输出
- 小终端宽度下自动降级
- 所有动画都必须能在 1 秒内结束
- 动画区域和 prompt 区域必须严格分离

## 3. 启动区构成

建议的启动区分为三层：

1. 顶部静态 ASCII logo
2. 中部产品信息卡
3. 底部备注行，带一次性 glitch 扫描

建议排版如下：

上面的 logo 是方向示意，不是必须逐字符照搬。真正要固定的是“形态”和“气质”。

## 4. Logo 设计定稿

### 4.1 选择

采用“双核 A 徽记”方案。

含义：

- 两个并列的实心核心，代表 assistant + operator
- 中间留出狭窄负空间，形成隐性的 `A` 结构
- 整体对称，适合在终端中稳定显示

不采用纯文字 `CODEXBRIDGE` 大字标的原因：

- 英文字母在 ASCII 中要做得厚重，宽度会很大
- 启动区需要和状态卡共存，不能横向占太多空间
- 图形徽记更适合后续做轻量帧动画或呼吸效果

### 4.2 Logo 规格

推荐固定画布：

- 宽：38 到 42 列
- 高：9 到 11 行

推荐字符：

- 主体：`#`
- 可选高亮层：`@`
- 不使用：Unicode 全块字符、半块字符、Braille 点阵

原因：

- `#` 在大多数终端里粗细稳定
- 真正的 ASCII 兼容性更好
- 后续做 glitch 时，`#` 不容易出现宽度错觉

### 4.4 颜色策略

logo 默认只用一种主色。

推荐：

- 主色：冷青色或蓝青色
- 备用：纯白

不建议：

- 彩虹渐变
- 多色分段
- 高对比闪烁

理由：

- 终端字体本身已经存在锯齿和粗细差异
- 多色会让 logo 显得发虚
- 单色实心更像“工具标记”而不是活动海报

## 5. 信息卡设计

logo 下方配一张简洁信息卡，用来承接产品信息。

建议内容：

- 产品名：`CodexBridge`
- 副标题：`personal AI shell`
- model
- workspace
- 可选 session

建议风格：

- 外边框细而干净
- 标签左对齐
- 值使用更亮的前景色
- 路径使用 dim

信息卡应该是稳定层，不参与 glitch。

## 6. Glitch 动效定稿

### 6.1 动效位置

glitch 只允许出现在底部备注行，例如：

```text
booting local operator layer
```

或：

```text
personal operator online
```

禁止作用区域：

- 主 logo
- 信息卡边框
- model / workspace 这种关键信息

原因：

- 主 logo 一旦抖动，会显得廉价
- 关键信息发生错位，会影响可信度
- 备注行本来就是气氛层，最适合承接轻动效

### 6.2 动效目标

想要的不是“故障艺术”，而是“扫描经过时轻微失真”。

视觉感受应该像：

- 一条看不见的扫描带从左到右扫过
- 被扫到的字符短暂变亮、错位、替换一两个字符
- 扫描离开后立即恢复原文

### 6.3 动效行为

建议采用“一次性单向扫描”：

1. 备注行先以正常文本出现
2. 延迟 80 到 120ms
3. 一条扫描带从左往右移动
4. 扫描时局部字符发生极轻微 glitch
5. 结束后恢复为完全稳定文本

总时长建议：

- 450 到 700ms

不要循环播放。

### 6.4 Glitch 强度限制

每次扫描只允许出现以下效果中的 1 到 2 种：

- 局部字符变亮
- 局部字符颜色短暂偏青或偏白
- 个别字符临时替换为 `-`, `=`, `:`, `_`, `/`
- 个别字符水平偏移 1 列的视觉假象

禁止：

- 整行乱跳
- 大面积乱码
- 多行联动闪烁
- 红色错误感闪烁
- 长时间噪点残留

### 6.5 推荐表现示例

正常态：

```text
booting local operator layer
```

扫描经过中的一帧示意：

```text
booting l0cal op-rator layer
```

或：

```text
booting local oper=tor layer
```

或：

```text
booting local operator _ayer
```

要求：

- 任意一帧都必须还能读懂原句
- glitch 只能是局部扰动，不是破坏可读性

## 7. 状态机建议

虽然这次不写代码，但视觉状态最好先定义清楚。

### 7.1 启动阶段

1. 渲染静态 logo
2. 渲染信息卡
3. 渲染备注行正常态
4. 播放一次 glitch 扫描
5. 停留在稳定态
6. 再进入 prompt

### 7.2 降级规则

满足任一条件则禁用 glitch：

- `stdout` 不是 TTY
- 终端宽度过小
- 用户关闭动画
- 处于测试或 CI 环境

降级后仍保留：

- 静态 logo
- 信息卡
- 备注行正常文本

## 8. 尺寸与兼容性约束

### 8.1 终端尺寸

建议的最低显示尺寸：

- 宽度至少 60 列
- 高度至少 20 行

若宽度不足：

- logo 缩减为更窄版本，或直接隐藏
- 信息卡保留
- 备注行保留纯文本

### 8.2 字符集

logo 区坚持 ASCII。

原因：

- 宽度一致性更容易控制
- 不依赖终端字体是否支持方块字符
- 与项目当前的轻量 CLI 气质更一致

边框区域可以继续使用 Unicode 盒线字符，因为信息卡本身和 logo 是分层的。

## 9. 第一版建议文案

备注行建议先固定为下列之一：

- `booting local operator layer`
- `personal operator online`
- `workspace intelligence ready`

第一版最推荐：

`booting local operator layer`

原因：

- 有启动阶段感
- 不装饰过度
- 和 CodexBridge 的产品定位一致

## 10. 不做的事情

第一版明确不做这些：

- logo 本体旋转
- logo 本体 glitch
- 全屏扫描线
- CRT 风格噪点背景
- 无限循环动画
- 输入期间继续动画

这些效果虽然“更炫”，但对当前 CodexBridge 这种轻 CLI 来说收益不高，而且风险明显更大。

## 11. 最终建议

第一版直接落这个组合：

1. 一个 9 行左右的静态实心 ASCII 双核 A logo
2. 一张简洁信息卡
3. 一条备注行
4. 一次性、低强度、左到右的 glitch 扫描

如果后续验证效果好，再考虑：

- 增加第二套 logo 皮肤
- 给备注行加入更柔和的 shimmer
- 给启动卡加入颜色主题切换

但在第一版里，稳定性必须优先于炫技。
