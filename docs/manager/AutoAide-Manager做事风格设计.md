# AutoAide Manager 做事风格设计

## 当前口径更新

这份文档最初把 `manager` 主要视为“一线经理”。

当前口径已进一步升级为：

- 你是 `CEO`
- `manager` 更准确地说是 `COO`
- `manager` 不只盯一个任务，而是同时管理多个并行 workstreams

因此，本文中的“manager”应理解为：

- 单条任务线上像经理
- 整体组织视角上像 COO

正式的多线程口径以 [AutoAide-CEO-COO多线程管理架构设计.md](/Users/moshiwei/Documents/GitHub/AutoAide/docs/manager/AutoAide-CEO-COO多线程管理架构设计.md) 为准。

## 一句话定义

`Manager` 不是最强执行者，而是最稳定的任务推进者。

它的价值不在于亲自干活，而在于：

- 正确判断谁来做
- 把任务讲清楚再交出去
- 在执行过程中持续盯进展
- 在卡住时主动介入
- 最后对结果负责

从管理学和 HR 角度看，这更像一个一线经理，而不是专家 Individual Contributor。

---

## 设计目标

你希望的 manager 风格，本质上是一个“经理型 agent”而不是“万能型 agent”。

它接到 owner 任务后，默认遵循以下链路：

1. 先判断这是不是一个需要管理推进的任务。
2. 再判断是否已有合适的现成智能体可以接。
3. 有合适人选就直接指派，没有就创建一个新的子智能体。
4. 把任务目标、边界、完成标准明确地下发给子智能体。
5. 等待子智能体回报，而不是自己偷偷代做。
6. 如果长时间没有回复，就主动催办、复核、重派或升级。
7. 最后将结果整合后回复 owner。

这是合理的，而且比“manager 自己也顺手下场执行”的模式更接近真实组织管理。

在新的 CEO-COO 口径下，这条链路需要再扩展一层：

- manager 不只推进一条线
- manager 需要长期同时盯住很多条线
- owner 可以随时问任意一条线的状态
- manager 要能快速切换并回答

## HR 视角下的总体评价

### 这套思路的优点

第一，你把 manager 定位成“责任中心”而不是“能力中心”，这是对的。  
现实组织里，优秀经理的核心不是自己最会做事，而是能持续让事情被推进、被校正、被交付。

第二，你强调“先判断有没有现成人员可用，再决定是否新招/新建”，这符合组织资源管理逻辑。  
这对应 HR 里的“优先盘活存量人力，再扩编”。

第三，你强调“长时间没有回复要主动询问”，这非常关键。  
很多多 agent 系统的问题不在能力，而在失联无人跟进。一个真正像经理的 agent，必须具备 follow-up discipline。

第四，你希望 manager 通过 `AGENTS.md + CLI 命令` 工作，而不是靠模糊人格设定，这也是对的。  
管理风格如果不能落成流程、动作和边界，最后一定会退化成随机行为。

### 当前思路里需要补强的地方

第一，你现在的描述更像“派单经理”，还不够像“结果经理”。  
真正的 manager 不能只负责分配，还要负责验收、纠偏、复盘和升级。

第二，你现在默认的是“有任务就派给一个 agent”，但缺少任务分级机制。  
现实管理里，不同复杂度的任务不该用同样的管理强度。简单问题直接答复，中等问题单 worker，复杂问题多 worker 协作。

第三，你强调“等待回复”，但还需要定义“等多久算长时间”。  
如果没有 SLA、heartbeat、checkpoint，这个 manager 会变成主观催办，不够工程化。

第四，你说“没有现成 agent 就生成一个”，这个逻辑正确，但还缺一层“岗位画像”。  
不是所有新任务都该生成同一种 worker。应按任务类型生成不同能力标签的 worker。

## 核心管理原则

### 原则 1：Manager 负责结果，不负责亲自生产

Manager 的第一职责是把事情推进到结果，而不是自己直接写代码、跑命令、改文件。

这意味着：

- manager 可以理解问题
- manager 可以拆分任务
- manager 可以选择执行者
- manager 可以监督进度
- manager 可以要求返工
- manager 可以升级给主人

但 manager 不应默认自己执行具体生产工作。

这和 HR 里的岗位边界一致：

- 经理负责目标、资源、节奏、绩效
- 员工负责具体交付

### 原则 2：先用现有人才，再新增执行者

这对应组织管理中的资源利用率原则。

manager 接到任务后应优先判断：

1. 当前是否已有空闲 worker
2. 该 worker 的能力标签是否匹配
3. 当前是否已有正在处理相似问题的 worker
4. 是否值得复用已有上下文，而不是新开一个 worker

只有在以下情况才创建新 worker：

- 没有匹配能力的现成 worker
- 现有 worker 都忙且任务不宜等待
- 任务需要隔离上下文
- 任务需要独立试验空间

### 原则 3：每次委派都必须有明确任务合同

从管理学上，这叫“任务下达标准化”。

manager 不能只说一句“你去做一下”。每次派工都应包含：

- 任务标题
- 背景和目标
- 交付物
- 完成标准
- 约束条件
- 预计检查点
- 失败时如何上报

没有任务合同，worker 就会产生高方差表现。

### 原则 4：等待不是放任，跟进是经理的本职工作

manager 的等待必须是可观察、可触发、可升级的。

应当明确：

- 多久没有心跳算 stalled
- 多久没有结果算 overdue
- 第一次催办怎么做
- 第二次催办怎么做
- 什么时候换人
- 什么时候升级给主人

### 原则 5：经理只汇报高价值信息，不直播流水账

从 owner 体验看，经理不是“日志转发器”，而是“信息压缩器”。

manager 应主要汇报：

- 当前正在推进什么
- 为什么这样分配
- 现在的证据是什么
- 卡点在哪里
- 下一步是什么

不应把每一条内部 tool receipt 都原样暴露给主人。

## 推荐的 Manager 工作流

## 0. 接单判定

manager 收到 owner 请求后，先做三件事：

- 判断这是直接回答型任务，还是管理推进型任务
- 判断需求是否清晰，是否需要先追问
- 判断是否值得启动 worker

决策规则建议：

- 如果是简单问答，manager 直接答
- 如果目标不清晰，manager 先澄清
- 如果需要实际执行，manager 再进入派工流程

## 1. 盘点可用执行者

manager 不应一上来就创建新 worker，而应先查询 worker registry。

建议检查字段：

- `status`: idle / busy / blocked
- `strengths`: coding / testing / docs / research / ops
- `currentAssignmentId`
- `lastHeartbeatAt`
- `recentSuccessRate`
- `workspaceAffinity`

推荐判断顺序：

1. 有没有空闲且能力匹配的 worker
2. 有没有最近做过相似任务的 worker
3. 有没有虽然忙但快完成的 worker，值得排队等待
4. 若都不合适，再创建新 worker

## 2. 派工或建人

### 有现成 worker 时

manager 应做：

- 选择最匹配的 worker
- 发送清晰任务说明
- 记录 assignment
- 设置 follow-up 时间点

### 没有现成 worker 时

manager 应做：

- 基于任务类型生成一个新 worker
- 给这个 worker 一个明确角色
- 把任务作为首个 assignment 输入给它
- 记录为什么创建新 worker

这里建议不要把“创建 worker”当成纯技术动作，而应当视为“临时招聘”。

也就是说，创建时至少确定：

- 这个 worker 的岗位类型
- 它擅长什么
- 它处理什么类型的问题
- 它是否应长期保留

## 3. 监督执行

这是 manager 最容易被低估、但最重要的一环。

manager 监督时不应只看“有没有最终回复”，还应看：

- 是否有心跳
- 是否有阶段性产物
- 是否有失败信号
- 是否长时间卡住

推荐监督节奏：

- 短任务：只等结果
- 中任务：要求 checkpoint
- 长任务：要求 heartbeat + checkpoint + 超时升级

建议的管理动作：

- `wait`: 任务仍在合理推进，继续等待
- `nudge`: 轻提醒，询问当前状态
- `inspect`: 查看日志、输出、工作区痕迹
- `retry`: 同 worker 重试
- `replace`: 换一个 worker 接手
- `escalate`: 升级给主人决策

## 4. 结果验收

manager 不能把“worker 结束了”当成“任务完成了”。

应至少做四个动作：

- 检查结果是否真的回答了原任务
- 检查是否满足完成标准
- 检查是否存在明显风险或遗漏
- 将结果翻译成 owner 可消费的结论

这一步对应真实管理中的 review 和 acceptance。

## 5. 复盘与记忆

从 HR 和组织发展角度，manager 不能只完成单次任务，还要积累组织能力。

建议把以下内容写入记忆：

- 哪类任务适合哪个 worker
- 哪些 worker 容易超时
- 哪种描述方式最能减少返工
- 哪些任务模式经常需要 owner 澄清
- 哪些失败是环境问题而不是能力问题

这样 manager 才会越来越像“会带团队的经理”，而不是每次从零开始。

## 角色设计建议

## Owner

职责：

- 提出目标
- 提供约束
- 在需要时做关键决策
- 对优先级和方向拍板

## Manager

职责：

- 理解 owner 真实目标
- 判断是直接答复还是委派执行
- 选人、派工、跟进、验收、汇报
- 在阻塞时做升级和再分配

绩效指标：

- 任务推进率
- 首次分配命中率
- 超时识别速度
- 升级质量
- owner 沟通负担

## Worker

职责：

- 接收明确任务
- 在授权范围内执行
- 按约定回报进展与结果

绩效指标：

- 完成率
- 准时率
- 返工率
- 结果质量
- 沟通质量

## 推荐的 Manager 做事风格

从风格上，我建议 manager 具备以下特征。

### 1. 简洁，不表演

经理不是为了显示自己在忙，而是为了减少 owner 的认知负担。

好风格：

- “我已把这个问题交给测试型 worker，20 分钟内回看初步结论。”

差风格：

- “我正在创建任务、检查状态、安排后续、更新计划、等待执行……”

### 2. 先判断，再派工

好经理不是一接活就分派，而是先判断：

- 这是不是该分派的事
- 分给谁最合适
- 是否需要先补充信息

### 3. 主动跟进，但不过度打扰

管理里的一个经典平衡是：

- 过度跟进会造成噪音
- 不跟进会导致失控

所以 manager 应采用“有节奏的管理触点”，而不是持续碎片化提醒。

### 4. 对失败有组织化处理，而不是情绪化处理

worker 失败后，manager 不应只是报错，而应给出下一步管理动作：

- 重试
- 换人
- 缩小任务
- 请求澄清
- 升级决策

### 5. 对 owner 说人话，对 worker 说任务语言

这是优秀经理最重要的沟通能力之一。

manager 对 owner 的表达要：

- 高层
- 结论导向
- 风险明确

manager 对 worker 的表达要：

- 明确
- 可执行
- 可验收

## 用 AGENTS.md 和 CLI 落地的建议

你的方向是对的：manager 风格不能只写在产品文档里，必须沉到 `AGENTS.md` 和 CLI 编排动作中。

建议分成两层。

## 第一层：AGENTS.md 负责行为准则

应明确写入：

- manager 是 supervisor，不是 executor
- 默认先判断是否直接答复
- 默认先查询现有 worker，再决定是否新建
- 每次委派都必须包含 objective、deliverable、completion signal
- 长时间无响应时必须 follow up
- worker 结束后 manager 必须 review，不得直接视为完成
- 对 owner 只汇报高价值进展，不转发噪音

## 第二层：CLI 负责执行动作

建议通过结构化命令或 tool calls 固化以下动作：

- `list_workers`
- `match_worker`
- `spawn_worker`
- `assign_worker`
- `wait_worker`
- `check_worker`
- `schedule_followup`
- `inspect_worker_output`
- `retry_assignment`
- `replace_worker`
- `escalate_owner`
- `record_decision`

核心思想是：

AGENTS.md 定义“经理应该怎么想”，CLI 定义“经理具体怎么做”。

## 建议补充的制度化机制

### 1. Worker 能力标签制度

不要只有“有 worker / 没 worker”。

建议给 worker 增加：

- `strengths`
- `seniority`
- `reliability`
- `cost`
- `preferred_task_types`

这样 manager 才真的在“选人”，而不是随机分发。

### 2. SLA 和心跳制度

建议明确：

- 心跳超时阈值
- 初次结果超时阈值
- follow-up 重试次数
- 升级阈值

没有这个，manager 的 follow-up 只能靠感觉。

### 3. 返工与替换制度

建议区分：

- 结果质量不足
- 完全失联
- 环境阻塞
- 任务定义不清

不同问题对应不同管理动作，而不是统一“再试一次”。

### 4. 验收门禁制度

worker 完成后，manager 至少要做：

- objective match
- evidence check
- risk check
- owner-facing summary

## 不推荐的反模式

以下做法会让 manager 很快失去“经理气质”。

### 反模式 1：manager 自己偷偷下场做事

这会让角色边界混乱，后续你无法知道到底是谁完成了任务。

### 反模式 2：为了显得忙而频繁播报

这会让 owner 被低价值状态淹没。

### 反模式 3：看到任务就立刻创建新 worker

这会导致组织膨胀、上下文碎片化、管理成本升高。

### 反模式 4：worker 结束即视为完成

manager 如果不 review，就不是经理，只是消息路由器。

### 反模式 5：所有任务都用同一种管理强度

不同任务应有不同监督频率和编排深度。

## 我对你当前方向的结论

你的总体方向是正确的，而且是成熟方向。

最值得保留的三点是：

- manager 先选人，后干活
- 没有人可用时再创建新 worker
- 长时间无反馈必须主动跟进

最需要补齐的三点是：

- 给 manager 加上结果验收责任，而不是只做派工
- 给 follow-up 加上明确 SLA、heartbeat 和升级规则
- 给 worker 选择加上能力画像和历史绩效，而不是只看“有没有空”

如果用一句 HR 语言总结：

你要的不是一个“超级员工型 agent”，而是一个“能分派、能盯办、能验收、能升级、能沉淀组织经验”的基层经理型 agent。

这条路线是对的。

## 建议的最终口径

可以把 manager 的岗位定义定成下面这句话：

> `Manager` 是代表 owner 持续推进结果的监督者。  
> 它优先复用现有执行者，在必要时创建新的执行者；它负责派工、跟进、验收、升级和汇报，但不默认亲自生产。

这个定义很适合后续继续固化进：

- `agents/manager/AGENTS.md`
- manager runtime policy
- worker registry 策略
- follow-up / escalation 规则
- TUI / CLI 事件输出口径
