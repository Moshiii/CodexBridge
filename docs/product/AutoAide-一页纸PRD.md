# AutoAide 一页纸 PRD

## 产品名称

AutoAide

## 产品定义

AutoAide 是一个数字副官。

它常驻在飞书、Telegram、WhatsApp、Slack、Web 等聊天入口中，代表 owner 持续接收任务、掌握上下文、调度 worker、跟踪进展，并把结果交付回来。

它不是执行器。

它是 owner 的信息中枢、协调中枢和任务推进中枢。

## 目标用户

第一阶段目标用户：

- 个人开发者 / 创业者
- 小团队 owner
- 需要长期跟进复杂任务的负责人
- 已经在聊天软件中组织工作的人

这些用户的共性是：

- 任务分散在多个对话里
- 上下文经常断
- 需要持续跟进，而不是一次性回答
- 想把事情交出去，但仍希望自己保持控制权

## 核心问题

用户今天已经有很多 AI 可以回答问题，但依然缺少一个能持续办事的角色。

主要问题包括：

- 事情交代完之后，没有人真正盯到底
- 多个人在同一个群里补充信息，AI 容易串线
- 任务跨天、跨线程后，上下文容易丢
- 用户需要重复解释背景
- 用户需要自己催进度、催结果、催回报
- 多个执行器即使能工作，也缺少统一控制点

## 核心价值

AutoAide 提供的价值不是“更聪明的回答”。

它提供的是：

- 一个持续在线的数字副官
- 一个代表 owner 盯事的 AI 角色
- 一个统一的任务和上下文控制点
- 一个能调度多个 worker 的管理中枢
- 一个会主动带着进展和结果回来的长期协作者

## 核心使用场景

### 场景 1：群聊内持续跟进任务

owner 把 AutoAide 拉进项目群，交代一个目标。

团队成员后续不断补充问题、约束、反馈。

AutoAide 需要：

- 记得不同人说过什么
- 不混淆不同任务线
- 把事情分派给 worker
- 在关键节点回来同步

### 场景 2：个人工作副官

owner 单独和 AutoAide 对话，把多个待办、研究、执行任务交给它。

AutoAide 需要：

- 帮 owner 维持优先级
- 跟踪长期任务
- 在任务卡住时主动提醒
- 在结果准备好时主动汇报

### 场景 3：多 worker 协同执行

同一个目标需要多个 worker 去执行不同部分。

AutoAide 需要：

- 理解任务拆分
- 为不同 worker 分配工作
- 跟踪各自进展
- 汇总成 owner 能理解的统一结果

## MVP 范围

第一版必须做到：

- owner 可以通过至少一种聊天入口和 AutoAide 对话
- AutoAide 能识别 owner、群聊、线程上下文
- AutoAide 能记录任务、状态、承诺和进展
- AutoAide 能把具体工作派给至少一种 worker executor
- AutoAide 能接收 worker 进展与结果
- AutoAide 能主动向 owner 回报 blocker、progress、result

## MVP 不做什么

第一版不做：

- 自己成为完整执行器
- 自己直接改代码 / 跑 shell / 做工程执行
- 大而全的 provider orchestration
- 一开始就支持所有聊天平台
- 一开始就做复杂组织权限系统

## 关键产品原则

### 1. 对 owner 忠诚

AutoAide 的默认立场必须永远站在 owner 一边。

### 2. 一个 owner，一个控制点

即使背后有多个 worker，owner 面对的也应该始终是一个清晰统一的 AutoAide。

### 3. 聊天只是入口，不是真相层

真正的真相层应该是任务、分派、承诺、进展和记忆。

### 4. AutoAide 负责管理，不负责执行

执行动作必须归属于具体 worker。

### 5. 主动汇报比被动回答更重要

数字副官的价值在于持续推进，而不是只在被问到时才回复。

## 成功指标

第一阶段最重要的指标不是模型分数，而是产品行为指标：

- 任务是否被持续跟进
- owner 是否减少重复解释
- 群聊内是否能正确区分不同人的任务线
- worker 结果是否能被稳定回收
- owner 是否能在不追问的情况下收到关键进展

## 当前工程映射

仓库中已经有这些基础模块：

- `manager-core`
- `task-system`
- `memory-system`
- `worker-orchestrator`
- `supervision-core`
- `owner-interface`
- `executor-codex`

这说明当前工程方向和产品定义基本一致：

- AutoAide 做 manager plane
- worker 做 execution plane

## 下一步

最优先的产品化任务：

1. 打通一个真实聊天入口
2. 打通一个真实 worker executor
3. 建立 owner -> AutoAide -> worker -> AutoAide -> owner 的闭环
4. 做出一个能稳定演示“数字副官”体验的端到端场景

## 对外讲述口径

对外表达时，最推荐的产品口径是：

- `AutoAide` 不是另一个更会回答问题的 AI
- `AutoAide` 是一个持续在线的数字副官
- 它不是直接执行者，而是 owner 的统一控制点
- 它通过 worker 团队推进执行，并主动带着进展和结果回来

如果要用一句话做 pitch：

> AutoAide 是一个常驻在聊天入口中的数字副官，代表 owner 持续接收任务、掌握上下文、调度 worker、跟踪进展，并把结果带回来。

---

## 宣传节奏建议

如果后续继续做对外传播，建议坚持这个顺序：

1. 先解释为什么“持续办事的 AI”是一个成立的新物种。
2. 再解释聊天入口、长期记忆、多 worker 协同这些需求为什么成立。
3. 最后再介绍 `AutoAide` 作为答案，而不是一开始就做竞品对比。

也就是说：

- 先解释趋势
- 再解释问题结构
- 最后再推出产品

这样更容易让外界理解 `AutoAide` 不是单纯的聊天机器人，也不是某个现有项目的替代品，而是一种更清晰的产品形态。
