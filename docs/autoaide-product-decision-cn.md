# AutoAide 产品决策稿

## 一、文档目的

这份文档用于重新明确 AutoAide 的核心产品决策。

这次决策的关键变化是：

- 不再把产品一开始硬拆成 `single_user` 和 `group` 两种模式
- 改为先建立统一的“身份识别模型”和“session 路由模型”
- 再基于不同上下文决定功能开放策略

这份稿子要解决的问题是：

- 无论消息来自群聊还是私聊，AutoAide 都必须知道“当前是谁在说话”
- 无论消息来自 Telegram、Feishu 还是未来其他 channel，AutoAide 都必须用统一规则路由会话
- `/goal`、schedule、stop、命令权限这些都应该建立在统一身份模型之上，而不是建立在粗糙的模式划分之上

## 二、产品总方向

AutoAide 应该被定义成：

- 一个多 Bot 的本地运行时系统
- 每个 Bot 只绑定一个外部 channel
- 所有 channel 都接入统一的身份识别和会话路由层

未来产品的一等公民不是：

- 单人模式
- 群聊模式

而应该是：

- 当前消息来自哪个 channel
- 当前消息属于私聊还是群聊
- 当前发消息的人是谁
- 当前这条消息应该路由到哪个 session

也就是说，产品的基础抽象要从“场景模式”改成“身份和上下文路由”。

## 三、核心决策

### 决策 1：一个 Bot 只绑定一个 channel

结论：

- 保留

原因：

- Bot 的身份、权限、接入配置、日志和运营边界都应保持单一
- 一个 Bot 同时挂多个 channel，会让记忆归属、身份投射和排障复杂度大幅上升

所以未来规则仍然是：

- 一个 Bot 对应一个 channel
- 需要 Telegram 和 Feishu 两个入口时，就创建两个 Bot

### 决策 2：统一身份识别模型是第一优先级

结论：

- 必须建立

AutoAide 在任何 channel 下都必须能够识别：

- 当前 query 是谁发来的
- 当前 query 是在私聊里还是在群里
- 当前 query 属于哪个会话容器

这是产品底层能力，不是可选增强。

### 决策 3：session 必须归属于“人”，而不是简单归属于“聊天容器”

结论：

- 必须如此

原因：

- 如果 session 只挂在 chat 上，群聊里所有人就会共用同一条记忆链
- 这会导致上下文污染、任务串台、stop 权限混乱、人格混乱

因此，AutoAide 必须先知道“谁在说话”，再决定“把这条消息接到哪条会话线上”。

### 决策 4：群聊/私聊的差异应体现在策略层，而不是基础身份层

结论：

- 群聊和私聊当然不同，但这种不同不应该先体现在产品主模型上
- 它应该体现在功能策略和权限策略上

也就是说：

- “识别是谁在说话”是统一底层能力
- “群聊里是否允许 `/goal`”是上层策略
- “私聊里是否要求 `@`”是上层策略
- “谁能 stop 谁的任务”也是上层策略

### 决策 5：一个 Bot 对外只暴露一条主会话，不开放手动切 session

结论：

- 应该如此

这里要区分“产品层”与“实现层”。

产品层面：

- 一个 Bot 对外只表现为一条连续主会话
- 不再把多 session 暴露给用户

实现层面：

- 系统内部仍然可以保留 session 机制
- 但 session 只作为内部路由与状态存储结构存在
- 由系统自动决定，不允许用户手动切换

原因：

- 用户不会自然地把聊天机器人理解成一个可以手动切换线程的终端工具
- 多 session 会严重破坏 Bot 的稳定人格和连续记忆感
- `/goal`、`/stop`、schedule、权限和状态提示都会因为“当前 session 是哪条”而复杂化
- 一旦已经采用按身份自动路由，用户层的 `/new`、`/switch`、`/sessions` 就失去产品必要性

因此正式决策是：

- 取消用户侧手动 session 管理能力
- 保留系统内部 session key，但把它降级为内部实现细节

## 四、统一身份模型

未来每个 channel adapter 都必须归一化出一套统一字段。

最少必须包含：

- `channel`
- `chatType`
- `chatId`
- `userId`
- `messageId`
- `isDirect`
- `isGroup`
- `explicitlyMentionedBot`
- `text`

推荐标准结构：

```json
{
  "channel": "feishu",
  "chatType": "group",
  "chatId": "oc_xxx",
  "userId": "ou_xxx",
  "messageId": "om_xxx",
  "isDirect": false,
  "isGroup": true,
  "explicitlyMentionedBot": true,
  "text": "@bot 你能做什么"
}
```

这里面最重要的是：

- `userId`
- `chatType`
- `chatId`

## 五、统一 chatType 归一化

内部统一只保留两种 chatType：

- `direct`
- `group`

映射规则例如：

- Telegram `private` -> `direct`
- Telegram `group` / `supergroup` -> `group`
- Feishu `p2p` -> `direct`
- Feishu 非 `p2p` -> `group`

这样做的目的是：

- 上层逻辑不再依赖各个平台自己的术语
- 以后扩 Slack、Discord、企业微信时也能复用同一套路由规则

## 六、统一 session 路由规则

这是本次决策最关键的部分。

### 6.1 私聊

私聊时，session 应该绑定到“用户”。

规则：

- `sessionKey = channel + userId`

解释：

- 私聊天然就是一个用户和 Bot 的一对一关系
- 会话归属不应该再挂在 chat 容器上，而应该直接挂在人身上

### 6.2 群聊

群聊时，session 也应该绑定到“用户”，但要额外带上群维度。

规则：

- `sessionKey = channel + chatId + userId`

解释：

- 同一个人在不同群里扮演的角色不同
- 不能让一个人跨多个群共享完全相同的上下文
- 但也不能让整个群共用一条 session

所以正确模型不是：

- 群一条 session

而是：

- 群里每个人一条 session

## 七、最终 session 归属规则

产品最终采用以下规则：

1. 私聊：
   - `sessionKey = channel + userId`

2. 群聊：
   - `sessionKey = channel + chatId + userId`

这条规则适用于：

- Telegram
- Feishu
- 未来所有 channel

但需要补充一条产品解释：

- 这里的 session 只是系统内部的会话归属键
- 不代表用户可以看见、切换或管理多个会话

## 八、群聊上下文模型

虽然群聊里的 session 应该归属于用户，但机器人不能忽略群现场。

因此未来 prompt 应采用双层上下文结构。

### 第一层：用户持久上下文

这是 session 本身的长期记忆。

包含：

- 这个用户和 Bot 以前聊过什么
- 这个用户在当前群里的持续上下文
- 这个用户最近的任务和话题

### 第二层：群现场上下文

这是每次 turn 的临时上下文。

包含：

- 最近若干条群消息
- 当前发言人是谁
- 当前消息在回复哪条
- 当前消息提到了谁
- 当前群的名字 / 标签

这两层结合之后，机器人会拥有：

- 对人的连续记忆
- 对群环境的即时感知

但不会把所有人的记忆混在一起。

## 九、命令与能力策略

这里要强调：

- 功能不是按“模式”硬切
- 而是按“上下文”决定是否开放

也就是说，能力控制规则应该写成：

- 在私聊里如何
- 在群聊里如何
- 在管理员身份下如何

而不是先说“这是 single_user mode，所以全部如何”

同时还要明确一条：

- session 不再是用户可操作对象
- session 只是系统自动路由的内部对象

## 十、`/goal` 的产品决策

### 10.1 私聊中的 `/goal`

结论：

- 保留

原因：

- `/goal` 本质上是个人长期目标监督能力
- 最适合发生在一对一关系里

### 10.2 群聊中的 `/goal`

结论：

- 默认不开放

原因：

- 群聊里目标归属不清
- 长流程监督会带来公共噪音
- stop、继续、完成判断都会复杂化

因此当前建议是：

- 私聊允许 `/goal`
- 群聊默认禁止 `/goal`

未来如果真要在群里启用，也只能是：

- 以管理员能力单独设计
- 或转入私聊继续执行

## 十一、schedule 的产品决策

### 11.1 私聊

结论：

- 保留

原因：

- schedule 本质上也是个人自动化工作流

### 11.2 群聊

结论：

- 默认不开放

原因：

- 群里 schedule 会带来归属、触发时机、通知噪音和权限问题

所以当前产品决策与 `/goal` 一致：

- 私聊保留
- 群聊默认禁止

## 十二、stop 的产品决策

### 12.1 私聊

- `/stop` 停止当前用户自己的当前任务

### 12.2 群聊

- `/stop` 默认只能停止“当前发言人自己”的任务
- 不允许默认停止别人的任务

未来如果要做管理员能力，可以追加：

- 管理员 stop 指定用户任务

但这不是当前第一阶段目标。

## 十三、session 用户能力决策

正式产品决策如下：

用户侧不再开放：

- `/new`
- `/switch`
- `/sessions`

原因：

- 这些命令是开发者工具心智，不是聊天产品心智
- 它们会让一个 Bot 看起来像多个线程容器，而不是一个稳定助手
- 一旦 session 已经按身份自动路由，用户手动管理 session 就会变成多余且危险的能力

因此未来产品原则是：

- 一个 Bot 对外只有一条主会话体验
- session 只在系统内部存在
- 系统自动根据身份与上下文决定路由到哪条内部会话

## 十四、并发与队列决策

### 13.1 私聊

建议规则：

- 每个用户一个活动任务
- 超过后直接提示 busy 或 queued

### 13.2 群聊

建议规则：

- 队列按用户 session 组织
- 不是按整个群统一一个总队列

原因：

- 群里每个人有自己的 session
- 那么执行管理也应该以用户为单位

这比“整个群一条队列”更合理，因为：

- 不会一个人把整个群卡死
- 不会让不同用户互相抢同一条 session

## 十五、`@` 触发策略决策

### 群聊

建议规则：

- 默认只在显式 `@机器人自己` 时回复

原因：

- 降低噪音
- 减少误触发
- 明确机器人被召唤的时刻

### 私聊

建议规则：

- 不要求 `@`

原因：

- 私聊天然就是用户和 Bot 的直接会话

## 十六、对 Midjourney / Discord 的参考结论

这次参考 Midjourney 在 Discord 上的设计，真正值得借鉴的不是它“群聊模式”的名字，而是它对“上下文”的处理方式。

核心启发有四点：

### 1. 平台上下文要明确

Discord 官方会区分：

- Guild
- Bot DM
- Private Channel

这说明：

- 平台上下文天然是产品一级概念

### 2. 公共空间和私有空间要区分能力

Midjourney 支持在公共频道中发起，也支持 DM 补充。

这说明：

- 群聊和私聊不应该做完全相同的事情
- 但这种差异应该体现在策略层，而不是破坏统一身份模型

### 3. 个人状态尽量不要污染公共频道

Midjourney 的 `/info` 更偏个人状态面板。

这说明：

- AutoAide 未来也应该尽量减少在群里输出长状态面板和系统噪音

### 4. 强命令边界在多人环境里更重要

Midjourney 的 slash commands 非常明确。

这说明：

- 多人共享环境下，控制类能力要更克制、更明确

## 十七、最终产品决策

AutoAide 下一阶段正式采用以下产品原则：

1. 一个 Bot 只绑定一个 channel

2. 任何 channel 都必须统一识别：
   - 当前是私聊还是群聊
   - 当前发言的人是谁

3. 所有 session 路由统一采用：
   - 私聊：`channel + userId`
   - 群聊：`channel + chatId + userId`

4. 群聊里 session 归属于用户，不归属于群本身

5. session 是内部实现细节，不作为用户能力暴露

6. prompt 采用双层结构：
   - 用户持久上下文
   - 群现场上下文

7. `/goal` 和 schedule 是上下文敏感能力
   - 私聊开放
   - 群聊默认不开放

8. `/stop` 在群聊里默认只作用于当前发言人自己的任务

9. 群聊默认只在显式 `@机器人自己` 时回复

## 十八、研发实施顺序建议

### 第一阶段：统一身份层

- 所有 channel adapter 输出统一 envelope
- 至少补齐：
  - `chatType`
  - `chatId`
  - `userId`
  - `messageId`
  - `explicitlyMentionedBot`

### 第二阶段：统一 session 路由层

- 新增共享 `resolveSessionKey(envelope)` 逻辑
- Telegram 和 Feishu 不再各自私自决定 session key

### 第三阶段：能力策略层

- `/goal` 改成只在私聊开放
- schedule 改成只在私聊开放
- `/stop` 在群聊改成“只能停自己的任务”
- 从用户侧移除 `/new`、`/switch`、`/sessions`

### 第四阶段：群现场上下文

- 为群聊补最近消息窗口
- 让 Bot 在每次回复时知道“刚刚群里发生了什么”

### 第五阶段：控制面与文案收口

- CLI 和 Web 都改成展示：
  - 当前 chatType
  - 当前 userId / actor
  - 当前 session 路由结果
- 不再把 session 作为用户可切换对象暴露

## 十九、最后结论

最终结论很明确：

- AutoAide 不应该先被定义成“single_user 产品”和“group 产品”
- 它应该先被定义成“能在任何 channel 里识别当前是谁在说话，并把消息路由到正确 session 的系统”

在这个基础上：

- 私聊和群聊再拥有不同的功能策略

同时：

- 一个 Bot 对外应该始终表现为一条主会话
- session 应降级为系统内部实现细节，而不是产品能力

这才是一个更稳定、更可扩展、也更适合多 channel 演进的产品架构。
