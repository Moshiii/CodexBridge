# CodexBridge 商业计划书

> 历史草稿。本文档早于当前按 bot 范围划分的运行时设计，仍包含守护进程时代的表述。已实现系统请参见 `docs/current-architecture.md`。

---

## 第 1 页：封面

### CodexBridge

面向已经在使用强模型、但厌倦了每次都要从零开始的人群的持久化个人 AI 操作员。

- 产品：本地优先的持久化 AI shell
- 当前形态：CLI + 守护进程 + Telegram
- 核心理念：给 Codex 一个“家”、记忆，以及始终在线的通道层

---

## 第 2 页：执行摘要

### CodexBridge 是什么

CodexBridge 将强模型从一次性的聊天工具，转变为持久化的个人操作层。

### 它解决了什么问题

今天，先进的 AI 工具很强大，但也是临时性的：

- 没有稳定的归属环境
- 没有持久的工作上下文
- 缺乏可长期运行的助手行为
- 跨会话与跨通道的连续性很弱

### 我们正在构建什么

- 位于 `~/.codexbridge` 的本地运行时主目录
- 位于 `~/.codexbridge/workspace` 的持久工作区
- 一个用于始终在线通道的单一守护进程
- 构建在 Codex CLI 之上的轻量编排层

### 商业判断

在以下两类产品之间，存在一个真实的产品品类：

- 原始聊天界面
- 重型企业级 agent 平台

CodexBridge 瞄准的正是这个中间地带：个人化、持久化、实用型的 AI 运营层。

---

## 第 3 页：问题

### 用户已经拥有模型能力，但没有产品连续性

强模型已经很适合用于：

- 编程
- 研究
- 写作
- 运营辅助

但当前的使用方式仍然存在问题：

- 每次新会话都会丢失本地操作上下文
- 有价值的偏好和记忆默认不能长期保留
- 跨通道交互是割裂的
- 缺少后台常驻能力
- 用户每次都得手动重新补全上下文

### 结果

模型虽然有用，但还不像一个可靠的助手。

---

## 第 4 页：为什么是现在

### 时机异常成熟

现在有三件事同时成立：

1. 前沿编码 agent 已经足够好，能在日常场景中真正发挥作用
2. 用户愿意把更多持续性工作委托给 AI
3. 当前的 UX 缺口已经很明显，而且确实令人痛苦

### 市场时机信号

下一波产品浪潮不只是“更好的模型”。
而是“更好的模型容器”。

CodexBridge 就是一个容器型产品：

- 持久
- 本地
- 具备通道感知能力
- 足够轻，能够快速迭代

---

## 第 5 页：产品愿景

### 产品目标

让强模型更像一个真正的个人操作员，而不是一个无状态的聊天会话。

### 理想用户体验

- 一个助手身份
- 一个持久工作区
- 一个存放长期记忆的位置
- 一个管理始终在线通道的守护进程
- 同一个助手可通过 CLI 和 Telegram 触达

### 设计哲学

- 本地优先
- 有意保持轻量
- 后端可灵活切换
- 用户可编辑
- 先追求实用，再追求“魔法感”

---

## 第 6 页：产品概览

### 当前已经具备的内容

- 通过 `codexbridge` 提供以 CLI 为主的入口
- 运行时主目录创建
- 工作区初始化
- 持久化 Markdown 上下文模型
- 基于 Codex 的执行与会话恢复
- 后台守护进程
- Telegram 桥接
- 会话路由
- stop/status/restart 控制

### 产品形态

```text
用户
  -> CLI / Telegram
  -> CodexBridge 守护进程 + shell
  -> 持久工作区
  -> Codex 运行时
```

---

## 第 7 页：这个产品为什么不同

### CodexBridge 并不试图成为一个庞大的 agent 平台

CodexBridge 没有重建一整套完整的 agent 运行时，而是有意保持轻量。

### 战略优势

- 复用前沿 agent 后端的能力
- 聚焦连续性、控制力和产品体验
- 相比完整编排栈，复杂度更低
- 后续可以切换或支持多个后端

### 定位

CodexBridge 不是：

- 另一个聊天机器人封装层
- 无代码企业工作流引擎
- 只面向内部使用的记忆插件

CodexBridge 是：

- 个人 AI 操作 shell

---

## 第 8 页：目标用户

### 核心用户

- 开发者
- 技术型创业者
- Codex / Claude Code / Gemini CLI 风格工具的高级用户
- 独立运营者
- 以终端 + 聊天工作流为主的研究者和知识工作者

### 早期采用者画像

这些人已经相信 AI 有用，但认为当前工作流太临时、太重复。

### 共同特征

- 对工具复杂度容忍度高
- 对重复劳动的痛感强
- 强烈需要连续性
- 愿意运行本地软件

---

## 第 9 页：用户价值主张

### CodexBridge 以三种方式节省时间

1. 更少的上下文重复
   - 助手能记住工作区状态和用户配置
2. 更少的工具摩擦
   - CLI、守护进程和通道访问被统一起来
3. 更低的会话重启成本
   - 用户持续与同一个助手协作，而不是每次都从零重新打开同一任务

### 情绪价值

这个助手会显得更沉稳、更可靠，也更像是“属于用户自己的”。

---

## 第 10 页：产品架构如何构成商业护城河

### 护城河不只是模型质量

模型质量来自上游，并且会随着时间推移不断被压缩。
真正持久的护城河，是模型周围的产品层：

- 持久工作区模型
- 本地运行时所有权
- 通道集成
- 会话连续性
- 用于 stop/status/restart 的控制层
- 可扩展的后端抽象

### 这为什么重要

随着后端不断进步，CodexBridge 无需自己发明智能，也能承接更多价值。

---

## 第 11 页：市场进入策略

### 第一阶段：由开发者驱动的采用

进入渠道：

- GitHub
- 原生终端分发
- 开发者社区
- AI 工具社区

### 增长循环

1. 用户安装 CodexBridge，因为他们本来就在使用强模型
2. 用户体验到持久连续性
3. 用户接入 Telegram，让助手长期在线
4. 用户分享工作流截图、演示或仓库
5. 其他高级用户跟进采用

### 为什么这条路径可行

这个产品天然适合演示，而且一旦看到实际效果，就很容易理解。

---

## 第 12 页：收入模式

### 短期收入选项

1. 开源核心 + 付费托管增值功能
   - 同步
   - 托管通道
   - 高级记忆 / 索引功能
2. 本地专业版订阅
   - 高级用户工作流
   - 更强的通道管理
   - 更好的持久化与自动化功能
3. 团队版
   - 共享助手
   - 共享策略 / 工作区模型
   - 管理控制能力

### 最现实的首个模式

先通过开源完成采用，再推出付费专业功能。

这与当前用户画像一致，也能降低早期分发阻力。

---

## 第 13 页：市场扩张路径

### 切入点

面向技术用户的个人持久化 AI 助手。

### 扩张路径

1. 更多通道
   - 先 Telegram，再扩展到其他消息触点
2. 更多后端
   - 先 Codex，之后支持 Claude Code / Gemini 风格后端
3. 更强的记忆和后台工作流
4. 共享与团队模式
5. 面向 operator 场景的自动化和审批系统

### 长期机会

占据个人 AI 操作层，而不只是首次使用时的那段对话。

---

## 第 14 页：竞争

### 竞争对手类别

1. 原始模型聊天产品
   - 很强，但无状态或持久性很弱
2. 编码 agent 与终端 agent
   - 执行能力强，但缺少个人产品壳层
3. 企业级 agent 平台
   - 笨重、昂贵、并非以个人场景优先
4. 记忆封装层与提示词层
   - 轻量，但通常浅层且脆弱

### CodexBridge 的优势

- 持久化本地工作区
- 始终在线的守护进程模型
- 实用的通道接入面
- 用户可编辑、可拥有的上下文
- 轻量架构叠加强后端杠杆

## 第 17 页：关键风险

### 产品风险

- 本地部署摩擦
- 后端依赖风险
- 长任务可靠性
- 通道边界场景的复杂性

### 市场风险

- 上游模型厂商自己增强持久化能力
- 早期阶段用户付费意愿仍不明确

### 缓解方式

- 保持轻量
- 快速迭代
- 优先服务痛点最强的用户
- 聚焦产品连续性，而不是试图在模型层面正面超越上游

---

## 第 18 页：财务框架

### 早期运营模型

这是一个低人数配置、产品驱动的软件型下注。

### 成本结构

- 小型工程团队
- 在开源 / 本地优先阶段，基础设施需求极少
- 相比企业级 agent 平台，固定运营成本更低

### 经济上行空间

如果 CodexBridge 成为技术用户每天都保持开启的持久化 shell，那么它的留存与变现能力都可能显著强于一次性 AI 工具使用。

## 第 20 页：结尾

### 核心陈述

AI 市场需要的不只是更好的模型。
它还需要更好的模型“居所”。

CodexBridge 正在构建这个居所：

- 持久化
- 个人化
- 本地优先
- 具备通道感知能力
- 逐步达到 operator 级别

### 最终结论

CodexBridge 可以成为用户整天保持开启的一层，而不是那个每次都要从零重新打开的工具。
